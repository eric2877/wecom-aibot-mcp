# 人工测试方案

> 针对无法自动化测试的模块，提供手动测试指南

---

## 一、config-wizard.ts 测试方案

### 1.1 交互式配置向导测试

**前置条件**:
- 已安装项目依赖
- 删除现有配置: `rm -rf ~/.wecom-aibot-mcp`

**测试步骤**:

```bash
# 1. 运行配置向导
npm run start

# 2. 观察输出，应显示配置向导
# 3. 按提示输入:
#    - Bot ID: 从企业微信管理后台获取
#    - Secret: 从企业微信管理后台获取
#    - 目标用户: 输入接收审批消息的用户账号
```

**预期结果**:
- [ ] 配置向导正确启动
- [ ] Bot ID 输入验证通过
- [ ] Secret 输入验证通过
- [ ] 用户 ID 验证请求发送成功
- [ ] 配置文件写入 `~/.wecom-aibot-mcp/config.json`
- [ ] Hook 脚本写入 `~/.wecom-aibot-mcp/permission-hook.sh`

### 1.2 用户 ID 自动检测测试

**前置条件**:
- 已配置 Bot ID 和 Secret
- 目标用户已添加机器人为好友

**测试步骤**:

```bash
# 1. 运行配置向导
npm run start

# 2. 选择"自动检测用户 ID"选项
# 3. 在企业微信中给机器人发送任意消息
# 4. 等待系统自动识别
```

**预期结果**:
- [ ] 显示"等待用户消息"提示
- [ ] 60 秒内收到用户消息
- [ ] 正确识别用户 ID
- [ ] 发送确认消息到企业微信

## 二、client.ts 测试方案

### 2.1 WebSocket 连接测试

**前置条件**:
- 已配置 Bot ID 和 Secret
- 网络连接正常

**测试步骤**:

```bash
# 1. 启动 MCP 服务
npm run start

# 2. 观察日志输出
# 3. 等待 WebSocket 连接建立
```

**预期结果**:
- [ ] 显示 "WebSocket 连接已建立"
- [ ] 显示 "认证成功"
- [ ] 心跳消息正常发送

### 2.2 重连测试

**测试步骤**:

```bash
# 1. 启动服务并确认连接成功
# 2. 断开网络连接（关闭 WiFi 或拔网线）
# 3. 等待 30 秒
# 4. 恢复网络连接
# 5. 观察日志输出
```

**预期结果**:
- [ ] 断线时显示 "连接断开"
- [ ] 自动尝试重连
- [ ] 重连成功后显示 "认证成功"
- [ ] 发送重连通知到企业微信

### 2.3 消息发送测试

**测试步骤**:

```bash
# 1. 进入微信模式
# 在 Claude Code 中说: "现在开始通过微信联系"

# 2. 观察企业微信是否收到消息
```

**预期结果**:
- [ ] 企业微信收到 "已进入微信模式" 消息
- [ ] 消息格式正确（包含 ccId）

### 2.4 审批卡片测试

**测试步骤**:

```bash
# 1. 在 headless 模式下执行敏感操作
# 例如: 删除文件、执行命令等

# 2. 检查企业微信是否收到审批卡片
# 3. 点击"允许"按钮
# 4. 观察操作是否执行
```

**预期结果**:
- [ ] 企业微信收到审批卡片
- [ ] 卡片显示操作描述
- [ ] 点击"允许"后操作执行
- [ ] 收到审批确认消息

## 三、connection-manager.ts 测试方案

### 3.1 多机器人连接测试

**前置条件**:
- 配置多个机器人 (至少 2 个)

**测试步骤**:

```bash
# 1. 添加多个机器人配置
# 编辑 ~/.wecom-aibot-mcp/config.json:
{
  "robots": [
    { "botId": "bot1", "secret": "secret1", "targetUserId": "user1" },
    { "botId": "bot2", "secret": "secret2", "targetUserId": "user2" }
  ]
}

# 2. 启动服务
npm run start

# 3. 进入 headless 模式，选择机器人
```

**预期结果**:
- [ ] 显示机器人选择列表
- [ ] 选择后正确连接
- [ ] 消息发送到正确的机器人

### 3.2 机器人占用检查测试

**测试步骤**:

```bash
# 1. 在终端 A 进入 headless 模式，选择机器人 A
# 2. 在终端 B 尝试选择同一个机器人 A
```

**预期结果**:
- [ ] 终端 B 显示 "机器人已被占用" 错误
- [ ] 终端 B 显示占用者信息

### 3.3 断线重连测试

**测试步骤**:

```bash
# 1. 启动服务并连接
# 2. 模拟网络断开
# 3. 等待自动重连
```

**预期结果**:
- [ ] 显示重连日志
- [ ] 重连成功后恢复正常

## 四、完整流程测试

### 4.1 端到端审批流程

**测试步骤**:

```
1. 用户: "现在开始通过微信联系"
2. Claude: 发送确认消息到企业微信
3. 用户: 在企业微信中回复 "执行 ls 命令"
4. Claude: 收到消息，执行命令
5. Claude: 发送执行结果到企业微信
6. 用户: "结束微信模式"
7. Claude: 确认退出，恢复终端交互
```

**预期结果**:
- [ ] 消息双向传递正常
- [ ] 审批流程正确触发
- [ ] 状态转换正确

### 4.2 多 CC 场景测试

**测试步骤**:

```
1. 在两个终端同时进入 headless 模式
2. 使用不同机器人
3. 在企业微信中分别给两个机器人发消息
4. 观察消息是否正确路由
```

**预期结果**:
- [ ] 消息正确路由到对应的 CC
- [ ] 无引用时显示提示消息
- [ ] ccId 正确显示在消息中

## 七、自动化测试命令

```bash
# 运行所有自动化测试
npm run test:coverage

# 运行 E2E 测试 (需要配置真实凭证)
RUN_E2E=true npm run test:e2e

# 运行特定测试文件
npm run test -- tests/unit/client.test.ts

# 监听模式开发
npm run test:watch
```
