# v1.3.0 规划：消息确认占位功能

## 背景

用户在企业微信发送消息后，需要等待 Claude 处理完成才能收到回复。
对于较长的处理任务（如代码生成、文件搜索），用户体验不佳。

**目标**：在消息被 Claude 消费后，立即推送"思考中"占位消息，让用户知道消息已被接收。

---

## 可行性分析

### 当前消息流程

```
用户消息 → WecomClient.handleMessage() → 消息队列
         → get_pending_messages() 返回 → Claude 处理
         → send_message() 推送结果
```

### 技术方案

#### 方案 A：简单占位消息（推荐）

**实现**：在 `get_pending_messages` 返回消息时，自动推送一条"思考中"消息。

**代码修改**：
```typescript
// src/tools/messaging.ts
server.tool('get_pending_messages', ..., async ({ ccId, ... }) => {
  const cached = client.getPendingMessages(clear);
  if (cached.length > 0) {
    // 新增：推送"思考中"占位
    await client.sendText(`【${ccId}】正在处理中...`);

    return { content: [...] };
  }
  // ...
});
```

**优点**：
- 实现简单，改动小
- 立即生效，用户体验提升明显

**缺点**：
- 占位消息和最终消息是两条独立消息
- 如果处理很快（<1秒），会看到闪现

#### 方案 B：企业微信原生 thinking 消息

**原理**：使用企业微信官方 thinking 消息类型，流式更新内容。

**前提条件**：
- @wecom/aibot-node-sdk 需支持 thinking 消息类型
- 需要查阅 SDK 文档确认 API

**调研需求**：
```bash
# 检查 SDK 支持的消息类型
npm info @wecom/aibot-node-sdk
# 或查看 node_modules/@wecom/aibot-node-sdk/types
```

**优点**：
- 原生支持，体验更好
- 消息更新而非追加

**缺点**：
- 依赖 SDK 支持
- 实现复杂度较高

---

## 推荐方案

**优先实现方案 A**，后续可升级为方案 B。

### 实现步骤（方案 A）

1. **修改 `src/tools/messaging.ts`**
   - 在 `get_pending_messages` 返回消息时调用 `sendText`
   - 可配置开关（默认开启）

2. **优化占位文案**
   ```
   【ccId】收到消息，正在处理中...
   ```

3. **可选：配置项**
   ```json
   // wecom-aibot.json
   {
     "enableThinkingPlaceholder": true,
     "thinkingPlaceholderText": "正在处理中..."
   }
   ```

### 后续优化（方案 B）

1. 调研 @wecom/aibot-node-sdk 的流式消息 API
2. 实现 thinking 占位 → 流式更新 → 最终消息
3. 参考 OpenClaw China 的实现
