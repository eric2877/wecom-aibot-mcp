## 2. 版本号体系混乱

项目同时存在 **4 套版本号**，无任何交叉说明：

| 版本号来源 | 示例值 | 用途说明 |
|-----------|--------|---------|
| `package.json` | v2.4.7 | NPM 发布版本 |

**建议**：制定版本号对照表，放在 `design/overview.md` 顶部：


## 3. 设计文档与代码严重不一致

### 3.1 `cc-registry.ts` 模块不存在

**文件**：[design/detailed-design.md §1.1](detailed-design.md#L1)

`detailed-design.md` 将 `cc-registry.ts` 列为独立模块：
```
src/
├── cc-registry.ts        # ccId 注册表管理
```

但实际 `src/` 目录中**无此文件**。ccId 注册表已并入 `src/http-server.ts`（L174 起的 `ccIdRegistry: Map<string, CCRegistryEntry>`）。

### 3.2 注册表实现与设计完全不同

`detailed-design.md §2.1` 描述了文件锁 + 原子写入的 `cc-registry.json` 方案：

```typescript
// 设计文档描述的实现
function registerCcId(ccId: string, robotName: string): boolean {
  const lockFile = `${CONFIG_DIR}/cc-registry.lock`;
  const fd = fs.openSync(lockFile, 'wx');  // 文件锁
  // ... 读写 cc-registry.json
}
```

**实际实现**（`http-server.ts:174`）是纯内存 Map，无文件持久化，无文件锁：

```typescript
// 实际实现
const ccIdRegistry = new Map<string, CCRegistryEntry>();  // 内存 Map
export function registerCcId(ccId, robotName, ...): { success: boolean; ccId: string } {
  ccIdRegistry.set(ccId, { robotName, agentName, mode, projectDir, lastOnline: Date.now() });
  return { success: true, ccId };
}
```

**影响**：
- 设计声明的"原子性"和"竞态条件防护"均未实现
- MCP Server 重启后，ccId 注册表丢失（内存清零）
- 架构文档中的 `~/.wecom-aibot-mcp/cc-registry.json` 文件在运行时并不存在

### 3.3 未文档化的实际模块

以下模块在 `src/` 目录中存在，但在任何设计文档中均未提及：

| 模块 | 用途（推断） |
|------|------------|
| `keepalive-monitor.ts` | 保活监控 |
| `client-pool.ts` | 客户端连接池（多机器人？） |
| `connection-log.ts` | 连接日志 |
| `src/channel-server.test.ts` | 测试文件（位于 src/ 而非 tests/，异常） |

`daemon.ts` 仅在旧设计文档中提到，与当前架构关系不明。

### 3.4 `detailed-design.md` 中的接口与实际不符

文档中 `registerCcId` 签名：
```typescript
function registerCcId(ccId: string, robotName: string): boolean;
```

实际 `http-server.ts:192` 签名：
```typescript
export function registerCcId(
  ccId: string, robotName: string, agentName?: string,
  mode?: 'channel' | 'http', projectDir?: string, isReconnect?: boolean
): { success: boolean; ccId: string }
```

返回类型从 `boolean` 变为对象，参数增加了 4 个，文档未更新。

## 4. 文档头部元数据格式不统一

16 个设计文档中存在 **5 种不同的头部格式**：

### 格式 A（overview.md / tools-api.md）
```markdown
> 更新日期：2026-04-14
> 架构版本：v3.1 - ccId 注册表 + 项目级配置 + 文档 MCP 代理
```
特点：无字段加粗，字段名中文无统一规范

### 格式 B（architecture.md / detailed-design.md）
```markdown
> **版本**：v2.4.0
> **更新日期**：2026-04-14
> **关联文档**：[architecture.md](architecture.md)
```
特点：字段加粗，有关联文档字段

### 格式 C（hook-approval-solution.md）
```markdown
> **日期**：2026-04-06
> **问题**：断线审批超时自动拒绝、审批消息路由问题
> **更新**：2026-04-17 - Channel 模式审批结果 SSE 推送修复
```
特点：面向问题驱动，字段含义与其他文档完全不同

### 格式 D（implementation-log.md）
```markdown
> **项目**：wecom-aibot-mcp v3.0 升级
> **开始日期**：2026-04-07
> **基准版本**：v1.2.0
> **目标版本**：v3.0.2
> **状态**：✅ 核心功能已完成
```
特点：项目日志专用字段

### 格式 E（hook-change-log.md / DESIGN.md）
无头部元数据块，直接正文

**建议统一格式**：

```markdown

> **更新日期**：YYYY-MM-DD
> **文档类型**：架构设计 | 详细设计 | API 参考 | 变更日志 | 问题记录
> **状态**：有效 | 已废弃 | 草稿
> **关联文档**：（如有）
```
